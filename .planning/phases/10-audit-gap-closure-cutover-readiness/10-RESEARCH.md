### Gap 3 — P0-7a: Mobile Location Selector Double-Tap Bug

**Current state:** Investigated `js/nav.js` lines 152–194. The behavior is intentional on narrow viewports: when `narrowPicker && slug && !isComingSoonLink`, `e.preventDefault()` stops default navigation and scrolls the info panel into view. A second tap would then fire the click again — but since `e.preventDefault()` was called on the first tap, the browser may queue a second synthetic click on mobile, causing double-toggle behavior.

**Root cause hypothesis:** On mobile, tapping a link triggers `touchstart` → `touchend` → `mousemove` → `mouseenter` → `click`. If the click handler calls `e.preventDefault()` but does NOT call `e.stopPropagation()`, the event may bubble to the overlay background handler (`locationOverlay.addEventListener('click', ...)` at line 145), which then calls `closeLocationOverlay()`. Net result: overlay opens, info-panel scrolls into view, overlay closes.

**Fix approach:** Add `e.stopPropagation()` after `e.preventDefault()` in the narrow-picker branch (line ~183). Also add a Playwright test with `isMobile: true` (using `devices['iPhone 14']`) that: taps a location link, asserts `#locationInfo` is visible, asserts overlay is still open.

**Wave placement:** Belongs in Wave 2 alongside 10-03 (a11y landmarks), since it requires a Playwright test addition and touches `js/nav.js`. Do NOT put in its own separate plan — the effort is small (1 line fix + 1 test) and the Playwright infrastructure is reused.

**GitNexus impact:** `js/nav.js` is a shared browser script. Run `gitnexus_impact({target: "nav", direction: "upstream"})` before editing. Risk level expected LOW-MEDIUM (leaf browser script, no Node callers).

### Gap 6 — P1-9: Per-Route Lang + Hreflang Cluster

**Current state:** `SiteLayout.astro` hardcodes `<html lang="en">` for all pages. Antwerp is Belgium (Dutch-speaking), so the page language should be `nl-BE`. Brussels (coming-soon) is bilingual but population is majority French-speaking; using `fr-BE` as a sensible default when/if the page gets language-specific content. [VERIFIED: data/locations.json read]

**hreflang topology analysis (CRITICAL finding):**

Google's hreflang spec is for alternate language/region versions of the SAME page content. `en-US` Houston and `nl-BE` Antwerp are NOT alternate versions of each other — they are different locations in different languages. [VERIFIED: Google Search Central docs fetch]

**Correct hreflang use case for this site:**
- IF Antwerp page had Dutch (`nl-BE`) and English (`en`) versions at two URLs: use hreflang pairs between those two URLs.
- For different US cities (all `en-US`): hreflang between them is NOT correct per Google spec (same language, same-ish content targeting different cities within one country → not a valid hreflang cluster use case).
- For Antwerp (`nl-BE`) ↔ any US location (`en-US`): hreflang is only appropriate if the pages are considered alternate-language versions of each other. A Belgian location page and a US location page are NOT the same page in different languages — they are different pages.

**Recommended approach for P1-9 (minimal scope, correct behavior):**

1. Fix `<html lang>` attribute:
   - Antwerp: `lang="nl-BE"`
   - Brussels: `lang="fr-BE"` (coming-soon, set for completeness)
   - All US locations and non-location pages: keep `lang="en"` (current)

2. Add `SiteLayout.astro` `lang` prop: `lang?: string` (defaults to `"en"`). Pass from location pages: `<SiteLayout lang={loc.lang ?? 'en'} ...>`.

3. Populate `hreflang` field in `data/locations.json` for Antwerp: `"hreflang": "nl-BE"`. US locations remain `"hreflang": null` because there is no valid hreflang cluster to form between US cities.

4. Do NOT emit `<link rel="alternate" hreflang>` cross-linking US ↔ Belgium unless the audit specifically requires it and a product decision confirms the pages are language-alternate versions. The audit finding may be about `<html lang>` being wrong, not about a full hreflang cluster. [ASSUMED: audit finding intent — verify with audit CSV before implementing hreflang links between different cities]

**If hreflang links between locations are required by the audit:** The only defensible hreflang cluster would be `antwerp` ↔ a future `en` version of the antwerp page (i.e., `x-default`). Do not link US city pages as hreflang pairs with Antwerp.

**Files touched:** `src/layouts/SiteLayout.astro` (add `lang` prop), `SiteHead.astro` (pass `lang` to `<html>`), `data/locations.json` (set `hreflang` field for Belgium locations), `antwerp.html` (legacy: add `lang="nl-BE"` to `<html>` tag if not migrating to Astro in this wave), `check-hreflang-cluster.js` (new validator — see Validation Architecture section).

**GitNexus impact:** `SiteHead.astro` is HIGH blast radius (every Astro page). Run `gitnexus_impact({target: "SiteHead", direction: "upstream"})` before editing. If `lang` only propagates to the `<html>` tag in `SiteLayout.astro` (not `SiteHead`), impact is contained to `SiteLayout`.

**dataLayer push pattern** (from web-vitals + GTM best practice):
```javascript
import { onLCP, onCLS, onINP } from 'web-vitals/attribution';
function sendToDataLayer({ name, delta, value, rating }) {
    window.dataLayer = window.dataLayer || [];
    window.dataLayer.push({
        event: 'web_vitals',
        web_vitals_name: name,
        web_vitals_delta: Math.round(delta),
        web_vitals_value: Math.round(value),
        web_vitals_rating: rating,
    });
}
onLCP(sendToDataLayer); onCLS(sendToDataLayer); onINP(sendToDataLayer);
```

### Gap 11 — P2-4: Cookie Banner / CMP

**Current state:** `SiteHead.astro` implements Consent Mode v2 defaults with profile-based defaults (`eu_strict` vs `us_open` vs `global_strict`) set before GTM loads. **The existing consent contract:**
- EU/Europe pages: default `analytics_storage: 'denied'`, `ad_storage: 'denied'`
- US pages: default `analytics_storage: 'granted'`, `ad_storage: 'granted'`
- The `wait_for_update: 500` on GTM gives 500ms for consent signal before GTM acts

**Cookie banner must:**
1. Read the current consent state from `window.__TM_CONSENT_STATE__` (or `window.dataLayer`).
2. Present accept/reject UI (at minimum for EU users; optionally for all users).
3. On user choice, call `gtag('consent', 'update', { analytics_storage: '...' })` and persist choice to `localStorage`.
4. Use `<aside aria-label="Cookie consent">` wrapper per audit requirement.
5. NOT break existing GTM consent-default initialization.

**Recommended library: `vanilla-cookieconsent` v3.x** (current: 3.1.0) [VERIFIED: npm view]. Reasons:
- Pure vanilla JS, no framework dependency — compatible with IIFE pattern used throughout this codebase.
- Built-in GTM/Consent Mode v2 integration via `onConsent` callback.
- Configurable modal vs bottom-banner UI.
- ~10KB minified.
- Actively maintained (last release 2025).

**Alternative considered:** Hand-rolled banner. Rejected — Consent Mode v2 lifecycle has many edge cases (Google's documented consent API has `wait_for_update`, `consent`, `update` command ordering). A battle-tested library handles these correctly.

**Alternative considered:** CookieYes or OneTrust (paid CMP). Rejected — overkill for 1 EU location + US locations; adds third-party script dependency.

**Integration contract with existing Phase 6 implementation:**

The banner `onAccept` callback should:

```javascript
gtag('consent', 'update', {

    ad_storage: 'granted',
    ad_user_data: 'granted',
    ad_personalization: 'granted',
});
window.__TM_CONSENT_STATE__ = Object.assign(window.__TM_CONSENT_STATE__ || {}, { analytics_storage: 'granted', ad_storage: 'granted' });
```

The banner `onReject` callback (EU users who click reject) should:
```javascript
// Consent already defaulted to 'denied' — no update needed unless user changes mind
```

**CSP impact:** `vanilla-cookieconsent` ships as a local JS file (no CDN dependency). `script-src 'self' 'unsafe-inline'` in `_headers` already permits it. No CSP change needed.

**Files touched:** `package.json` (add `vanilla-cookieconsent@3`), new `js/cookie-consent.js` (IIFE wrapper around library init), `css/` (cookie banner styles — can use library's default CSS or override), `src/components/SiteScripts.astro` (load cookie-consent.js after GTM init), `SiteHead.astro` (no change needed — existing consent defaults are compatible).

## Tooling Decisions

### 1. axe-core Runner: `@axe-core/playwright` vs `pa11y-ci`

**Decision: `@axe-core/playwright`**

**Rationale:** Playwright is already installed at `^1.59.1`. `@axe-core/playwright` current version is `4.11.3` (wrapping `axe-core 4.11.4`). Adding it requires only one new devDependency. The existing `playwright.config.js` starts `python3 -m http.server 4173` as the local server — `check-img-alt-axe.js` can use the same server pattern, or serve from `dist/` using `astro preview`. [VERIFIED: npm registry]

**Rejected: `pa11y-ci`** — would add a separate headless Chrome binary and configuration, creating redundancy with existing Playwright setup.

**Rejected: standalone `axe-core` + `puppeteer`** — `puppeteer` is not installed; adding it means another binary.

### 2. web-vitals Version and API

**Decision: `web-vitals@5.2.0` attribution build**

Import from `web-vitals/attribution`. Use `onLCP`, `onCLS`, `onINP` (INP replaced FID as a Core Web Vital in March 2024). [VERIFIED: npm view confirms 5.2.0 is latest]

The script should be an IIFE in `js/web-vitals-rum.js` consistent with the project's browser script pattern. Initialize only if `window.__TM_CONSENT_STATE__.analytics_storage === 'granted'` (US pages) or after consent grant on EU pages.

### 3. hreflang Topology: Antwerp Only, No Cross-City Clusters

**Decision: Fix `<html lang>` for Antwerp (`nl-BE`) and Brussels (`fr-BE`). Do NOT emit hreflang links between US cities or between US ↔ EU location pages.**

**Rationale:** Google hreflang is for alternate-language versions of the same page. Different city pages (Houston vs Antwerp) are not alternates of each other — they are separate pages. Emitting spurious hreflang links between them would be incorrect implementation and could confuse Google. [VERIFIED: Google Search Central docs]

**What `check-hreflang-cluster.js` validates:** Parse all built `dist/*.html` files, assert:
- `antwerp.html` has `<html lang="nl-BE">` (not `lang="en"`).
- `brussels.html` (if it becomes an Astro page in this phase) has `<html lang="fr-BE">`.
- All US location pages have `<html lang="en">`.
- No location page emits cross-city hreflang pairs.

### 4. WordPress Redirect Status: 301 Only

**Decision: Use 301 (already implemented). Do NOT attempt 410.**

**Rationale:** Cloudflare Pages `_redirects` supports only 301, 302, 303, 307, 308. HTTP 410 is NOT in the supported set. Attempting `410!` syntax would either silently fail or produce a parse error. [VERIFIED: Cloudflare Pages docs]

The existing implementation (301 to `/`) is correct for Cloudflare Pages. Document this limitation explicitly in `docs/cutover-checklist.md` under "WordPress-era paths".

### 5. Validator Approaches

**`check-schema-altname`:** Already built into `check-schema-output.js` lines 148–178. No new script needed. Run `npm run check:schema-output` post-build. The validator name `check-schema-altname` described in CONTEXT.md is essentially what the existing validator already does.

**`check-img-alt-axe`:** New Playwright + `@axe-core/playwright` script. Runs against built `dist/`.

**`check-tap-targets`:** New static CSS audit script. Runs against CSS source files.

**`check-hreflang-cluster`:** New HTML parser script. Runs against built `dist/*.html` files and asserts `<html lang>` attribute values match `data/locations.json` region/hreflang fields.

## Cookie Banner / CMP Strategy (P2-4)

**Recommendation: `vanilla-cookieconsent` v3.1.0**

Integration contract with existing Phase 6 Consent Mode v2:

```javascript
// js/cookie-consent.js (IIFE, loaded after GTM init in SiteScripts.astro)
(function() {
    'use strict';
    // Only show banner if current consent profile is eu_strict or global_strict
    // US open users have analytics_storage: 'granted' by default — no banner needed
    var profile = window.__TM_TAGGING_CONFIG__ && window.__TM_TAGGING_CONFIG__.consent_profile;
    if (profile === 'us_open') return; // no banner for US-defaulting pages
    
    CookieConsent.run({
        onFirstConsent: function({ cookie }) {
            // Update Consent Mode v2
            if (cookie.categories.analytics) {
                gtag('consent', 'update', { analytics_storage: 'granted', ad_storage: 'granted' });
            }
        },
        // ... categories, language, UI config
    });
})();
```

**CSP impact:** None — `vanilla-cookieconsent` ships as `js/cookie-consent-lib.js` (local file), served from `'self'`. Existing `script-src 'self' 'unsafe-inline'` already allows it.

**Why show the banner only for EU visitors?** `SiteHead.astro` already sets `consentProfile` based on the page's location region. Antwerp and Brussels pages get `eu_strict` defaults. The banner should mirror this logic: only present on EU-region pages (or `global_strict`). US visitors already have `granted` defaults per Phase 6 design.

**Tap-target compliance for banner:** The banner's accept/reject buttons must meet ≥44px height. `vanilla-cookieconsent` v3 default CSS meets this. If customizing, verify in `check-tap-targets.js`.
