# Phase 10 — UI Design Contract

> Visual and interaction contract for the four new/modified surfaces in Phase 10:
> cookie banner, legal page chrome, skip-link focus state, and hero `<picture>` srcset.
> Plus conventions for decorative SVG aria-hidden and tap-target minimums.
>
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.
>
> **PRIMARY CONSTRAINT:** Visual parity is the design contract. The current static site
> is the locked reference. This phase does NOT redesign, re-palette, re-type, or introduce
> a new component library. Every surface below must feel native to the existing Time Mission
> dark sci-fi brand.

Phase 10 surfaces must map to these existing tokens; no new custom spacing values.

| Token | Value | CSS Variable | Usage |
|-------|-------|-------------|-------|

## Surface Contracts

### Surface 1: Cookie Banner (P2-4)

**Library:** `vanilla-cookieconsent@3.1.0`
**Scope:** EU-region pages only (`eu_strict` consent profile) — Antwerp and future Brussels.
**Absent on:** all US-region pages (Philadelphia, Houston, Mount Prospect, West Nyack, Lincoln, Manassas, Orland Park, Dallas).

**Gate logic (JS):**
```javascript
// Read from window.__TM_TAGGING_CONFIG__ set by SiteHead.astro Phase 6 consent init
var profile = window.__TM_TAGGING_CONFIG__ && window.__TM_TAGGING_CONFIG__.consent_profile;
if (profile === 'us_open') return; // no banner
```

**Position:** Bottom-center modal on first visit. Rationale: minimizes layout distraction
on first paint while remaining clearly visible. Avoid bottom-right slide-in — the Time Mission
brand uses full-width bottom elements (ticker bar, mobile CTAs) and a slide-in would collide
visually on mobile. The bottom-center modal pattern is less intrusive than a full overlay.

**Layout:**
```
┌───────────────────────────────────────────────────┐
│  We use cookies                                   │
│  [description — 1–2 sentences]                    │
│                                                   │
│  [Accept all]   [Reject all]   Customize          │
└───────────────────────────────────────────────────┘
```

**Mobile:** Full-width sheet pinned to viewport bottom. `width: 100%; border-radius: 0` at
`max-width: 600px`. No horizontal scroll permitted.

**Wrapper element:** `<aside aria-label="Cookie consent">` — required by VALIDATION.md.
`vanilla-cookieconsent@3` renders its own DOM; configure the library's `guiOptions` to
use `position: 'bottom center'` and `layout: 'box'`.

**localStorage key:** `tm_consent_v1` — set as `vanilla-cookieconsent`'s `cookie.name`
config option. This makes the key transparent and environment-versioned. Example:
```javascript
CookieConsent.run({
    cookie: { name: 'tm_consent_v1' },
    ...
});
```

**Categories (maps to Consent Mode v2):**

| Category | Default | Consent Mode v2 signal |
|----------|---------|------------------------|
| `necessary` | always on | always granted |
| `analytics` | off | `analytics_storage` |
| `marketing` | off | `ad_storage`, `ad_user_data`, `ad_personalization` |

**onConsent / onFirstConsent callbacks:**
- Accept all → `gtag('consent', 'update', { analytics_storage: 'granted', ad_storage: 'granted', ad_user_data: 'granted', ad_personalization: 'granted' })`
- Reject all / customize with analytics=off → no update needed (EU defaults already `denied` from Phase 6 SiteHead init)
- Update `window.__TM_CONSENT_STATE__` to mirror the gtag update

**Focus management:** The banner must be focusable via keyboard. The "Accept all" button
receives focus when the banner appears (library handles this if configured). Tab order:
Accept all → Reject all → Customize. All buttons must have `:focus-visible` outlines using
`outline: 2px solid var(--orange); outline-offset: 3px` — consistent with `css/nav.css`
focus pattern.

**Tap-target compliance:** All three buttons must meet the 44px minimum height contract
from this spec's Tap Target section below. Use `min-height: 44px; padding: 0.75rem 1.5rem`.

**CSS override strategy:** Load `vanilla-cookieconsent`'s default CSS, then override color
variables in a `css/cookie-consent.css` override file loaded after. Override only:
- `--cc-bg` → `var(--dark)` (#151515)
- `--cc-primary-color` → `var(--orange)` (#FF6B2C)
- `--cc-btn-primary-bg` → `var(--gradient-primary)` (gradient on accept button)
- `--cc-btn-secondary-bg` → `transparent`
- `--cc-btn-secondary-border-color` → `rgba(255,255,255,0.25)`
- `--cc-text` → `var(--white)`
- `--cc-secondary-text` → `var(--gray)`
- `--cc-font-family` → `var(--body)` (DM Sans)

**Re-open mechanism:** Footer "Cookie Preferences" link calls `CookieConsent.showPreferences()`.
The footer link must use `<button>` or `<a href="#">` with `role="button"` — match existing
footer link style (`color: var(--gray); font-family: var(--body); font-size: 0.85rem`).

### Surface 3: Skip-Link Focus State (P1-3)

**Context:** `css/base.css` already contains the full `.skip-link` ruleset (confirmed by file
read at lines 59–76). The CSS is complete and correct. No CSS changes are needed for the
skip-link visual state.

**Current CSS (already in `css/base.css` — do not duplicate or override):**
```css
.skip-link {
    position: fixed;
    top: 1rem;
    left: 1rem;
    z-index: 20000;
    transform: translateY(-150%);
    padding: 0.75rem 1rem;
    border: 2px solid var(--orange);
    border-radius: 999px;
    background: var(--black);
    color: var(--white);
    font-weight: 700;
    text-decoration: none;
    transition: transform 0.2s ease;
}

.skip-link:focus {
    transform: translateY(0);
}
```

**Behavior contract:**
- Default: off-screen via `transform: translateY(-150%)` — invisible to sighted users
- On keyboard focus (`:focus`): slides into view at `top: 1rem; left: 1rem` — `position: fixed`
  ensures it appears above the ticker bar and nav regardless of scroll position
- Transition: 0.2s ease transform — smooth slide, not instant pop. This is already defined.
- `z-index: 20000` — above all overlays (nav z-index is 1000, location overlay is 10001)
- `border-radius: 999px` — pill shape, consistent with `.btn-tickets` pill style
- Color contrast: `var(--white)` on `var(--black)` = 21:1 — exceeds WCAG AA requirement

**Note on `:focus` vs `:focus-visible`:** The existing rule uses `:focus` (not `:focus-visible`).
This is intentional — the skip-link should show for ALL focus events (including mouse click)
because its purpose is to provide a keyboard-accessible target. `:focus-visible` would hide it
for programmatic focus which is incorrect for this pattern. Keep `:focus` as-is.

**SSR implementation contract:**
The `<a class="skip-link" href="#main">Skip to main content</a>` must be the FIRST child
of `<body>` in `SiteLayout.astro`, before `<slot name="pre-nav" />`. Example:
```html
<body class={bodyClass} data-location={bodyDataLocation}>
    <a class="skip-link" href="#main">Skip to main content</a>
    <slot name="pre-nav" />
    ...
    <main id="main" tabindex="-1">
        <slot />
    </main>
    ...
</body>
```

**Target element:** `<main id="main" tabindex="-1">` — the `tabindex="-1"` allows
programmatic focus on the element when the skip-link is activated. This is the SSR
landmark added in Plan 10-03.

**Impact on `js/a11y.js`:** The `ensureSkipLink()` function checks
`if (document.querySelector('.skip-link')) return` on line 5. Once the SSR skip-link
is in the DOM, this guard fires immediately and the JS injection becomes a no-op.
No functional change to `js/a11y.js` is needed — add a comment noting the guard
was originally the primary injection path and is now a safety net.

**Semantic landmarks required in `SiteLayout.astro`:**

| Landmark | Element | Attributes | Wraps |
|----------|---------|------------|-------|
| Banner | `<header>` | `role="banner"` | `pre-nav` slot + `nav` slot (ticker + nav + location overlay) |
| Main | `<main>` | `id="main" tabindex="-1"` | default slot |
| Content info | `<footer>` | `role="contentinfo"` | `footer` slot |

`<TicketPanel />` and `<SiteScripts />` remain inside `<body>` after `<footer>` — they
are not wrapped in a landmark. This is correct: TicketPanel is a `dialog`-role overlay;
SiteScripts emits only `<script>` tags.

### Surface 4: Hero `<picture>` srcset (P1-18)

**Context:** The Philadelphia location partial (`src/partials/philadelphia-main.frag.txt`)
uses a `<video>` element for its hero, not a `<img>`. (Confirmed by file read — the hero
section is a `hero-video-container` with a `<video id="heroVideo">` element, not an image.)

This means P1-18's srcset work applies only to Astro-rendered location pages where the hero
is a still image. The executor must audit each location partial to determine which use
`<img>` vs `<video>`. Houston must also be checked. If both Philadelphia and Houston use
`<video>`, P1-18 may have zero `<picture>` candidates among currently Astro-rendered pages.

**If a location partial uses `<img>` (the srcset target):**

Replace:
```html
<img src="/assets/photos/venue/hero.jpg" alt="Time Mission [City]">
```

With:
```html
<picture>
    <source
        srcset="/assets/photos/venue/hero-1920w.jpg 1x, /assets/photos/venue/hero-3840w.jpg 2x"
        media="(min-width: 1024px)"
    >
    <source
        srcset="/assets/photos/venue/hero-768w.jpg 1x, /assets/photos/venue/hero-1536w.jpg 2x"
        media="(max-width: 1023px)"
    >
    <img
        src="/assets/photos/venue/hero-1920w.jpg"
        alt="Time Mission [City]"
        width="1920"
        height="1080"
        loading="eager"
        fetchpriority="high"
    >
</picture>
```

**Breakpoints:**
- Desktop (≥ 1024px): 1920×1080 @ 1x; 3840×2160 @ 2x
- Mobile (< 1024px): 768×432 @ 1x; 1536×864 @ 2x
- Breakpoint 1024px aligns with the existing nav hamburger breakpoint in `css/nav.css`

**Format:** JPEG only — matches existing asset format in `assets/photos/`. AVIF/WebP sources
are deferred to a future optimization pass. Do not add WebP sources in Phase 10.

**Aspect ratio:** 16:9 (1920×1080) — lock with `width` and `height` attributes on `<img>` to
prevent CLS. Do not use `aspect-ratio` CSS property (legacy browser concern not needed here
since the HTML attributes achieve the same browser behavior for CLS prevention).

**Loading behavior:** `loading="eager" fetchpriority="high"` — hero is above the fold; LCP
element must not be deferred. This is required for Largest Contentful Paint score.

**Alt text:** `"Time Mission [City]"` — descriptive, matches brand name pattern. Never empty.
Never decorative for a hero image (it conveys location identity).

**Asset availability caveat:** If the location's hero asset does not exist at the required

## Decorative SVG aria-hidden Convention

**Rule:** Every inline SVG in Astro partials and components that is purely decorative (arrows,
icons, scroll indicators, stat backgrounds, separator graphics, social icons when adjacent to
visible text) must carry both `aria-hidden="true"` and `focusable="false"`.

**Pattern:**
```html
<!-- Decorative SVG — no information value -->
<svg aria-hidden="true" focusable="false" width="16" height="16" viewBox="0 0 24 24" ...>
    ...
</svg>
```

**Exception — informational SVG:** An SVG that conveys meaning without adjacent text (e.g.,
a standalone logo in a context where no visible text identifies it, a status indicator) must
NOT get `aria-hidden`. Instead it gets a `<title>` element as the first child:
```html
<svg role="img" aria-labelledby="svg-title-id">
    <title id="svg-title-id">Time Mission logo</title>
    ...
</svg>
```

**Application scope for Phase 10:**
- `src/partials/index-main.frag.txt` — arrow icons in CTAs, stat section icons, scroll
  indicators, swipe indicators (all decorative): add `aria-hidden="true" focusable="false"`
- `src/partials/philadelphia-main.frag.txt` and `houston-main.frag.txt` — arrow in
  `.btn-primary` CTA (decorative — button text conveys meaning): add `aria-hidden="true" focusable="false"`
- Logo SVGs (`.nav-logo img`, footer logo): these are `<img>` elements with `alt` text,
  not inline SVGs — no change needed
- Location overlay pin icon (`.location-btn svg`): decorative — the button has `aria-label`
  on the button element itself: add `aria-hidden="true" focusable="false"` to the SVG
- Close button X icon (`.location-dropdown-close` child SVG): decorative — button has
  `aria-label="Close location selector"`: add `aria-hidden="true" focusable="false"`

**Partial author guidance (for future partials):**
> When adding an inline `<svg>` to a partial, ask: "Does removing this SVG cause a user to
> lose information?" If no: add `aria-hidden="true" focusable="false"`. If yes: add a `<title>`
> and `role="img"` with `aria-labelledby` instead.

## Tap Target Minimums (P1-1)

**Contract:** Static CSS audit via `scripts/check-tap-targets.js`. The script checks CSS
source files (not computed runtime values) against known selectors.

**Minimum heights:**

| Selector group | Minimum height | Rationale |
|---------------|---------------|-----------|
| Nav interaction buttons | 44px | WCAG 2.5.5 success criterion |
| Social icon links (mobile menu) | 44px | WCAG 2.5.5 |
| Location overlay close button | 44px | WCAG 2.5.5 |
| Booking CTAs (`.btn-primary`, `.btn-tickets`) | 48px | Higher-value conversion action |
| Cookie banner buttons | 44px | WCAG 2.5.5 minimum |

**Selectors the validator (`check-tap-targets.js`) must audit:**

```
# 44px minimum — nav interaction elements
.nav-menu-btn          (min-height: 44px — CONFIRMED in css/nav.css media max-width:1024px)
.location-btn          (min-height: 44px — CONFIRMED in css/nav.css media max-width:768px)
.location-dropdown-close (width: 44px; height: 44px — CONFIRMED in css/nav.css)
.mobile-menu-socials a (width: 44px; height: 44px — CONFIRMED in css/nav.css)

# 44px minimum — general interactive
.btn-secondary         (check: padding must sum to ≥44px with line-height)
.mobile-menu-links a   (check: padding 1.1rem 0 — line-height + padding sum must ≥44px)

# 48px target — booking CTAs
.btn-tickets           (check: padding 0.75rem 1.5rem — must confirm height ≥48px)
.btn-primary           (check: must confirm height ≥48px)
.mobile-menu-cta a     (padding 1rem — line-height must sum to ≥48px with standard body line-height)
```

**Known current state from CSS read:**
- `.nav-menu-btn` at `max-width: 1024px`: `min-width: 44px; min-height: 44px` — PASSES
- `.location-btn` at `max-width: 768px`: `min-width: 44px; min-height: 44px` — PASSES
- `.location-dropdown-close`: `width: 44px; height: 44px` — PASSES
- `.mobile-menu-socials a`: `width: 44px; height: 44px` — PASSES
- `.btn-tickets` at `max-width: 768px`: `min-height: 44px` — PASSES (at 44px not 48px;
  desktop uses `padding: 0.75rem 1.5rem` = 12px×2 = 24px + line-height — executor must
  measure desktop computed height)
- `.mobile-menu-cta a`: `padding: 1rem` — 16px×2 = 32px + font line-height; DM Sans at

**If a selector fails the audit:** The fix is a CSS addition to `css/nav.css` or `css/base.css`

**Validator scope:** `css/base.css` and `css/nav.css` only. The cookie banner buttons are
